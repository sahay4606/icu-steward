1. EXECUTIVE SUMMARY
================================================================================

ICU Steward 2.0 is a multi-tenant SaaS platform designed for ICU operations
management with a focus on antibiotic stewardship, investigation tracking,
patient management, and clinical workflow coordination. It is built as a Yarn
v1 workspaces monorepo containing an Expo Router (React Native) frontend and
an Express.js backend backed by Supabase (PostgreSQL).

STRENGTHS:
  - Clean domain separation and consistent repository pattern on backend
  - Well-designed design system with centralized theme tokens
  - Good database schema with proper indexes and foreign keys
  - Multi-tenant architecture via hospital_id scoping
  - Every interactive element has data-testid for testing
  - ESM modules throughout (consistent module system)
  - Seed data is realistic and comprehensive for demos

CRITICAL WEAKNESSES:
  - ZERO authentication on 50+ CRUD API endpoints (CRITICAL)
  - Unsalted SHA-256 password hashing instead of bcrypt/argon2 (CRITICAL)
  - Service Role Key exposed — full database admin bypassing RLS (CRITICAL)
  - Mass assignment vulnerability — raw req.body passed to DB (CRITICAL)
  - Open CORS with no origin restrictions
  - Hardcoded JWT secret fallback in env.js
  - Error messages leak internal details to clients
  - Orphaned dead code (server.js) that references nonexistent files
  - Services layer fully instantiated but entirely unused in request pipeline
  - No CI/CD, no Docker, no tests, no monitoring, no logging infrastructure
  - No pagination on any list endpoint
  - Hardcoded ACTIVE_HOSPITAL_ID and ACTIVE_USER_ID in frontend config
  - Full data refetch on every mutation (no optimistic updates, no caching)
  - Notification-router expects columns that don't exist in schema
  - Role config keys (admin/doctor) mismatch seed data roles (Hospital Admin/ICU Consultant)

VERDICT: This codebase is at an early ALPHA / MVP prototype stage. It
demonstrates good architectural intentions and scaffolding but has fundamental
security, operational, and engineering gaps that make it NOT production-ready.
It is suitable for demo/trial use only. Estimated effort to reach production
readiness: 3-6 months with a dedicated team of 3-4 engineers.

3.4 DATA FLOW (Request Lifecycle)

  User Action → Screen Component → useData() hook
    → Context calls api.post/patch/delete
    → API client (fetch) → HTTP → Express Route
    → buildRepoRoutes → repository.insert/update/remove
    → Supabase JS (service_role) → PostgreSQL
    → Response → normalizeKeys (snake→camel)
    → refetchAll() → re-fetches ALL entities
    → Context state update → Re-render

  Login Flow:
    Login Screen → authService.login(email, password)
      → POST /api/auth/login → SHA-256(password) → DB lookup
      → JWT signed with secret → response {user, token}
      → Token stored in memory Map (native) or localStorage (web)
      → AuthContext state updated → AuthGate redirects to tabs

7. REQUEST/DATA FLOW ANALYSIS
================================================================================

7.1 FETCH ALL DATA ON APP START

  DataContext useEffect:
    1. Fires 9 parallel GET requests:
       - getHospitalSettings(hosp-st-john)
       - getPatients(hosp-st-john)
       - getInvestigations(hosp-st-john)
       - getAntibiotics(hosp-st-john)
       - getDevices(hosp-st-john)
       - getTasks(hosp-st-john)
       - getNotifications(hosp-st-john)
       - getUsers(hosp-st-john)
       - getReminderRules(hosp-st-john)
    2. Then GET /api/dashboard?hospital_id=hosp-st-john
    3. Then GET /api/timeline-events?patient_id=<id> for first 3 patients
       (sequentially, not parallel)
    4. All responses normalized via normalizeKeys (snake→camelCase)
    5. Results stored in DataContext state object

  PERFORMANCE CONCERN: All data for all patients loaded at once on app
  initialization, regardless of which tab is visited. No lazy loading.
  Full timeline fetched for first 3 patients only (arbitrary limit).

7.2 CREATE/UPDATE/DELETE FLOW

  User action → Context method (e.g., createPatient)
    → api.post('/api/patients', body with hospital_id)
    → Backend repository.insert(payload)
    → refetchAll() — re-fetches 6 entities in parallel
    → Context state updated → Re-render

  PERFORMANCE CONCERN: Every single mutation triggers a full data reload of
  all entities. No optimistic updates. No granular cache invalidation.

7.3 AUTH FLOW

  App start:
    AuthContext useEffect → storage.getItem('@icu_auth_user', '@icu_auth_token')
    → GET /api/auth/me (validates token)
    → If valid: sets user + token in state → AuthGate renders tabs
    → If invalid: clears storage → AuthGate renders login

  Login:
    LoginScreen → authService.login(email, password)
    → POST /api/auth/login
    → Backend SHA-256 hashes password
    → SELECT * FROM users WHERE email = ?
    → Compares hash
    → Returns user + JWT (7-day expiry)
    → Token stored in memory Map (native) / localStorage (web)
    → User stored as JSON string in same storage
    → AuthGate redirects to /(tabs)

  SECURITY CONCERN: Token stored in memory Map on native (lost on app restart
  unless secureGet/secureSet used — which they are NOT). AuthContext uses
  plain storage.getItem/setItem, not secureGet/secureSet.

7.4 DASHBOARD DATA FLOW

  GET /api/dashboard?hospital_id=xxx
  Backend fires 5 parallel Supabase queries:
    1. All patients → filter non-Stable → count
    2. Pending investigations → count
    3. Antibiotics with 'Review due' or 'High alert' → count
    4. Unacknowledged notifications → count
    5. Patients admitted in last 48 hours → count

  Also returns hardcoded 0 for positiveCultures and deviceReviewDue.

8. DATABASE ANALYSIS
================================================================================

8.1 SCHEMA OVERVIEW (14 tables in backend/supabase/master.sql)

  hospitals           (id: text PK)
  users               (id: text PK, FK→hospitals)
  patients            (id: text PK, FK→hospitals, 5 JSONB columns)
  investigations      (id: text PK, FK→hospitals, FK→patients)
  antibiotics         (id: text PK, FK→hospitals, FK→patients)
  devices             (id: text PK, FK→hospitals, FK→patients)
  tasks               (id: text PK, FK→hospitals, FK→patients nullable)
  notifications       (id: text PK, FK→hospitals)
  timeline_events     (id: text PK, FK→hospitals, FK→patients)
  reminder_rules      (id: text PK, FK→hospitals)
  daily_checklists    (id: text PK default gen_random_uuid, FK→hospitals, FK→patients)
  audit_logs          (id: text PK default gen_random_uuid, FK→hospitals)
  sessions            (id: text PK default gen_random_uuid, FK→users)
  role_assignments    (id: text PK default gen_random_uuid, FK→hospitals, FK→users)

8.2 SCHEMA STRENGTHS
  - Proper foreign keys on all tables with REFERENCES constraints
  - 28+ indexes covering hospital scoping, created_at, foreign keys, status
  - All tables have created_at + updated_at timestamps
  - Timestamptz used consistently (timezone-aware)
  - pgcrypto extension for gen_random_uuid()
  - RLS enabled on all tables (good foundation even without policies)
  - patients JSONB columns allow flexible clinical data without schema changes
  - Cascade-ready foreign key relationships

8.3 SCHEMA WEAKNESSES

  CRITICAL:
  - notifications table MISSING columns expected by service layer:
    - No source_user_id (notification-router.js line 12)
    - No assigned_doctor_id (notification-router.js line 13)
    - No patient_id (notification-router.js line 7)
    - No channel (notification-router.js line 15)
    - These inserts will silently drop extra keys or fail

  - audit_logs service passes { action, payload } but schema expects:
    - entity_type (NOT NULL, no default)
    - entity_id (NOT NULL, no default)
    - performed_by
    - details (jsonb, not 'payload')
    - The service will FAIL to insert every time

  - patients table stores denormalized JSONB arrays:
    - current_antibiotics, pending_investigations, completed_investigations,
      devices, daily_checklist are JSONB copies of data in dedicated tables
    - This creates dual-write risk — data in JSONB can diverge from
      authoritative tables (investigations, antibiotics, devices, daily_checklists)
    - No triggers or application logic to keep them in sync

  HIGH:
  - No CHECK constraints on status fields (e.g., status could be any string)
  - No ENUM types for status, priority, severity, role — all free-text
  - Text primary keys (e.g., 'p-101', 'u-admin') — human-readable but
    no auto-generation for production (gen_random_uuid used only in 4 tables)
  - Text PKs could cause fragmentation and slower joins vs UUIDs
  - No ON DELETE CASCADE on foreign keys — deleting a hospital will fail
    due to child record references
  - No unique constraint on uhid (patient identifier that should be unique)

  MEDIUM:
  - Migration files are not numbered for sequential execution
    (00001_create_tables.sql and master.sql overlap significantly)
  - master.sql is essentially the full migration + seed in one file
  - seed.sql doesn't include email/password_hash columns (added in migration 001)
  - No migration runner/versioning — must be manually pasted into Supabase UI

8.4 INDEX ANALYSIS

  Adequate coverage. Every table has an index on hospital_id and on foreign
  keys. Status, priority, date, and acknowledgment columns are indexed where
  they're used in WHERE clauses. No missing indexes identified for current
  query patterns.

8.5 PERFORMANCE CONCERNS
  - The patients JSONB columns could cause table bloat (each row stores
    duplicated investigation/antibiotic/device data)
  - No partial indexes on status/priority filters (but acceptable at this scale)
  - Sequential timeline fetch for first 3 patients (frontend side inefficiency)

  - No field selection (always SELECT *)

  - No rate limiting headers
  - No versioning in URL path (though acceptable at this stage)
  - No OPTIONS/CORS preflight consideration

9.3 REQUEST VALIDATION

  - auth/signup validates presence of name, email, password, hospitalId
  - auth/login validates presence of email, password
  - dashboard validates hospital_id query param
  - profile validates user_id query param
  - CRUD endpoints: ZERO validation — req.body and req.query passed directly
    to database

9.4 API DOCUMENTATION

10.2 STATE MANAGEMENT

  POSITIVES:
  - React Context pattern is appropriate for this scale
  - AuthContext + DataContext separation is logical
  - useMemo used to derive filtered/sorted lists
  - useCallback for memoized functions
  - Cancellation flag prevents state updates after unmount

  NEGATIVES:

  - No optimistic updates — user sees loading state on every action
  - No error boundaries — a single failed API call can break the entire data load
  - dashboardSummary in promise chain — if dashboard fails, all other data
    is still loaded (good), but error is swallowed (catch → null)
  - Timeline events fetched sequentially for first 3 patients only — partial data

10.3 COMPONENT ARCHITECTURE

  POSITIVES:
  - Clean separation of atomic UI components (pill, surface, field, etc.)
  - Consistent design system via theme.js
  - Every interactive element has data-testid
  - Surface component provides consistent card styling
  - SectionHeader pattern for consistent section layout
  - Timeline component with nice GitHub-style visualization

11. BACKEND ANALYSIS
================================================================================

11.1 ARCHITECTURE QUALITY

  The backend has a well-intentioned layered architecture:
  - config/ — environment, table names, role definitions
  - db/ — database client, repository factory, backend wiring
  - domain/ — per-entity repository factories
  - middleware/ — reusable Express middleware
  - modules/ — route definitions grouped by domain
  - services/ — business logic services

  However, the architecture is BROKEN BY DESIGN:
  - Services are instantiated in createBackend() (backend.js) but NEVER used
    in any route module. Routes either use raw supabase client (auth, dashboard,
    profile, settings) or generic repositories (all CRUD routes).
  - The services/auth.js wraps Supabase Auth methods, but auth.routes.js
    handles JWT manually and does its own DB queries.
  - The access-control service exists with `can()` and `requireTenantScope()`
    but no route ever calls them.
  - The notification-router service has detailed parameter structure, but
    notifications are created directly via repository in route handlers.

  This means: ~200 lines of service code is DEAD CODE that creates a false
  sense of security/architecture.

11.2 REPOSITORY PATTERN

  The generic CRUD repository (db/repository.js) is clean and consistent:
  - list, getById, insert, update, remove methods on every entity
  - Tenant scoping built-in via hospital_id filter
  - Dynamic query parameter filtering
  - Ordered results

  However:
  - No pagination on list()
  - No soft-delete support (remove does hard DELETE)
  - insert/update accept raw payloads with zero validation
  - No request-scoped transaction support
  - list() accepts arbitrary filter keys from query params — could be abused
    to filter on internal columns

11.3 MIDDLEWARE

  auth-middleware.js: Clean JWT verification. Only used on 1 route.
  async-handler.js: Standard Express async error wrapper. Good.
  error-handler.js: Returns err.message verbatim. Information leak.
  build-repo-routes.js: Clean CRUD route builder. No auth integration.

11.4 DOMAIN FILES

  All 14 domain files are one-liner wrappers:
    export function createXxxRepository(client) {
      return createRepository(client, BACKEND_TABLES.xxx, { ...options });
    }

  This is boilerplate that could be auto-generated from a config array.
  No custom business logic, computed fields, or validation in any domain file.

11.5 ORPHANED DEAD CODE

  backend/src/server.js (155 lines):
  - Imports from ./lib/supabase.js and ./lib/backend.js — FILES THAT DON'T EXIST
  - Contains inline auth logic with x-user-id header (old pattern)
  - Contains duplicate route building logic
  - Would crash immediately if imported or executed
  - This file should be DELETED

12. SECURITY AUDIT
================================================================================

12.1 CRITICAL — NO AUTHENTICATION ON CRUD ENDPOINTS

  Severity: CRITICAL (10/10)
  Files: backend/src/modules/*/ (all 10 CRUD route modules)
  Evidence:
    - buildRepoRoutes (middleware/build-repo-routes.js) creates GET/POST/PATCH/DELETE
      routes with zero auth checks
    - app.js mounts these routes without any auth middleware wrapper
    - auth-middleware.js exists but is ONLY applied to GET /api/auth/me
    - This means ANYONE can read/write/delete ALL patient data, user accounts,
      settings, etc. without any authentication
  Impact: Complete data breach. Anyone who knows the API URL can access all
    hospital data, modify patient records, delete data, create users, etc.

12.2 CRITICAL — UNSALTED SHA-256 PASSWORD HASHING

  Severity: CRITICAL (10/10)
  File: backend/src/modules/auth/auth.routes.js, line 33, 47
  Evidence:
    - const hash = crypto.createHash('sha256').update(password).digest('hex');
    - No salt, no iterations, no bcrypt/scrypt/argon2
    - SHA-256 is a fast hash not designed for passwords
    - Seed data hashes can be cracked instantly (admin123, consult123, resident123)
    - Rainbow tables would reverse most common passwords immediately
  Impact: Complete compromise of all user passwords if database is breached.
    Attackers can authenticate as any user.

12.3 CRITICAL — SERVICE ROLE KEY EXPOSURE

  Severity: CRITICAL (9/10)
  File: backend/src/db/client.js, line 7
  Evidence:
    - Uses SUPABASE_SERVICE_ROLE_KEY (not anon key)
    - Service role bypasses ALL Row Level Security policies
    - .env file containing actual keys is committed to repository
  Impact: Anyone with access to the .env file (committed in git) has full
    admin access to the Supabase database. RLS is effectively useless.

12.4 CRITICAL — MASS ASSIGNMENT VULNERABILITY

  Severity: CRITICAL (9/10)
  File: backend/src/middleware/build-repo-routes.js, lines 14-16, 19-21
  Evidence:
    - router.post('/', asyncHandler(async (req, res) => {
        const data = await repo.insert(req.body); ...
    - router.patch('/:id', asyncHandler(async (req, res) => {
        const data = await repo.update(req.params.id, req.body); ...
  Impact: Any user (authenticated or not) can set any column on any entity.
    For users table: can set role='admin', password_hash, status, etc.
    For patients: can set hospital_id to access other tenants' data.

12.5 HIGH — NO INPUT VALIDATION

  Severity: HIGH (8/10)
  File: backend/src/middleware/build-repo-routes.js (entire file)
  Evidence:
    - req.body passed directly to database operations
    - No schema validation, no type checking, no sanitization
    - req.query used directly as filters — could enumerate columns
  Impact: Malformed or malicious payloads reach the database. Could
    potentially exploit PostgreSQL or Supabase vulnerabilities.

12.6 HIGH — OPEN CORS

  Severity: HIGH (7/10)
  File: backend/src/app.js, line 24
  Evidence:
    - app.use(cors()) — no origin whitelist, allows all origins
  Impact: Any website can make authenticated API calls from a browser if
    user is logged in. Enables CSRF-style attacks.

12.7 HIGH — ERROR INFORMATION LEAKAGE

  Severity: HIGH (7/10)
  File: backend/src/middleware/error-handler.js, line 3
  Evidence:
    - res.status(500).json({ error: err.message });
    - err.message can contain SQL errors, stack traces, internal paths
  Impact: Attackers can probe endpoints and learn internal system details
    from error messages.

12.8 HIGH — HARDCODED JWT SECRET FALLBACK

  Severity: HIGH (8/10)
  File: backend/src/config/env.js, line 14
  Evidence:
    - jwtSecret: env[BACKEND_ENV_KEYS.jwtSecret] || 'icu-steward-dev-secret'
  Impact: If JWT_SECRET is missing from environment, falls back to a known
    string. Anyone who reads this code can forge valid JWT tokens.

12.9 HIGH — NO RATE LIMITING

  Severity: HIGH (7/10)
  File: Entire backend — no rate limiting middleware anywhere
  Impact: Login endpoint vulnerable to brute force. All endpoints vulnerable
    to DoS. API abuse possible without throttling.

12.10 HIGH — JWT SECRET READ AT MIDDLEWARE CREATION TIME

  Severity: HIGH (6/10)
  File: backend/src/middleware/auth-middleware.js, line 5
  Evidence:
    - jwtSecret read once during factory call, not per-request
    - If env changes, old tokens still validate with old secret
  Impact: No way to rotate JWT secret without restarting server.

12.11 HIGH — NO REQUEST SIZE LIMITS

  Severity: HIGH (6/10)
  File: backend/src/app.js, line 25
  Evidence:
    - app.use(express.json()); — no limit parameter
  Impact: Large payload attacks can exhaust server memory.

12.12 HIGH — LIVE CREDENTIALS IN .env COMMITTED

  Severity: HIGH (9/10)
  File: backend/.env (live Supabase credentials committed to repository)
  Impact: All credentials (SUPABASE_URL, ANON_KEY, SERVICE_ROLE_KEY, JWT_SECRET)
    are in the git history. Anyone with repo access has full database access.

12.13 MEDIUM — AUTH TOKENS NOT IN SECURE STORE (NATIVE)

  Severity: MEDIUM (5/10)
  File: frontend/src/contexts/AuthContext.js, lines 18-19
  Evidence:
    - Uses storage.getItem/setItem (in-memory Map on native)
    - secureGet/secureSet exist in storage adapter but are NOT used
  Impact: Auth tokens persist only in memory on native (lost on app restart)
    or in localStorage on web (no encryption).

12.14 MEDIUM — SIGNUP HARDCODES ROLE

  Severity: MEDIUM (5/10)
  File: backend/src/modules/auth/auth.routes.js, line 36
  Evidence:
    - Role hardcoded to 'Senior Resident' on signup
  Impact: All new users get the same role regardless of their actual position.

12.15 MEDIUM — ROLE/SEED DATA MISMATCH

  Severity: MEDIUM (5/10)
  File: backend/src/config/roles.js vs master.sql
  Evidence:
    - Config keys: admin, doctor, nurse, pharmacist, consultant, resident
    - Seed data roles: 'Hospital Admin', 'ICU Consultant', 'Senior Resident'
    - No mapping between these sets
  Impact: RBAC cannot match database roles to permission definitions.

12.16 MEDIUM — NOTIFICATION SCHEMA MISMATCH

  Severity: MEDIUM (5/10)
  File: backend/src/services/notification-router.js lines 5-15 vs master.sql
  Evidence:
    - Service expects: patient_id, source_user_id, assigned_doctor_id, channel
    - Schema has: id, hospital_id, title, detail, severity, time, acknowledged
    - Service inserts will silently ignore extra keys
  Impact: Notifications created via the service will lose critical metadata.

12.17 MEDIUM — AUDIT LOG SCHEMA MISMATCH

  Severity: MEDIUM (5/10)
  File: backend/src/services/audit-log.js lines 4-8 vs master.sql
  Evidence:
    - Service passes { action, payload }
    - Schema requires: action, entity_type, entity_id, performed_by, details
    - Missing: entity_type (NOT NULL), entity_id (NOT NULL), performed_by
    - Key name mismatch: payload vs details
  Impact: The auditLog.record() function will FAIL on insert every time.

12.18 LOW — NO CSRF PROTECTION

  Severity: LOW (3/10)
  Evidence: No CSRF tokens, no SameSite cookie config, no anti-CSRF headers
  Mitigation: API is stateless (Bearer token), which mitigates CSRF somewhat.

14. SCALABILITY REVIEW
================================================================================

14.1 DATABASE SCALING

  - Supabase PostgreSQL can scale vertically and with read replicas
  - Current schema has no partitioning strategy
  - Text primary keys are less efficient for large-scale joins than UUIDs
  - JSONB columns on patients table cause table bloat and slow queries
  - No materialized views or query optimization for dashboard aggregates

14.2 API SCALING

  - Express.js single-threaded — needs clustering or PM2 for multi-core
  - No rate limiting — single user could saturate the API
  - No caching — every request hits the database
  - No compression (gzip) on responses
  - Synchronous request handling per connection (no queue/worker pattern)

14.3 FRONTEND SCALING

  - All data loaded on mount — not viable beyond ~1000 patients
  - No virtualized lists (FlatList instead of SectionList/VirtualizedList)
  - No paginated data loading
  - Everything in a single bundle with no code splitting

14.4 MULTI-TENANT SCALING

  - hospital_id scoping is correct for tenant isolation
  - However: no connection pooling per tenant, no tenant-aware caching
  - All tenants share the same database (fine for most SaaS, but no isolation
    guarantees)

23. FINAL VERDICT
================================================================================

ICU Steward 2.0 is an EARLY-STAGE PROTOTYPE with the skeleton of a
well-architected product. The codebase demonstrates:

  - Good architectural intentions (layered backend, repository pattern,
    clean front-end design system, multi-tenant data model)
  - Working prototype features (patient management, antibiotic stewardship,
    investigations, tasks, notifications, dashboard)
  - Realistic and well-constructed seed data for demos

However, the codebase has FUNDAMENTAL FLAWS that prevent any production
deployment:

  1. COMPLETE LACK OF AUTHENTICATION on all data-access endpoints
     (50+ CRUD endpoints are publicly accessible)

  2. CRITICAL SECURITY VULNERABILITIES (unsalted SHA-256 passwords,
     mass assignment, service role key exposure, hardcoded JWT secret,
     open CORS, error information leakage)

  3. BROKEN ARCHITECTURE (service layer instantiated but entirely
     disconnected from the request pipeline — auth, RBAC, audit logging,
     and notification routing are all non-functional)

  4. NO OPERATIONAL INFRASTRUCTURE (no CI/CD, no Docker, no monitoring,
     no logging, no testing, no backup strategy)

The codebase would require an estimated 3-6 months of intensive engineering
effort (3-4 person team) to reach production readiness. The architectural
foundation is solid enough to build upon, and the domain model is well
thought out. But critical security and operational gaps mean this should
NOT be deployed in any real healthcare environment today.

HEALTHCARE SPECIFIC CONCERN: For a product handling patient data, the
security gaps are particularly alarming. HIPAA compliance, data encryption,
audit trails, and access controls are minimum requirements that this
codebase completely fails to meet.
