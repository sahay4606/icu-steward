# ICU Steward Rollout Plan

## Goal
- Replace mock data with Supabase-backed data.
- Add login and role-based access.
- Deploy web and ship iOS/Android apps.
- Support different access levels for doctors, nurses, admins, and future roles.
- Add alerting so nurse actions can notify doctors when needed.

---

## What We Built So Far (End-to-End Working)

### 1. Database and Migrations
- Tables + seed data in `backend/supabase/master.sql` — run once in Supabase SQL Editor.
- Auth columns in `backend/supabase/migrations/001_add_auth_columns.sql`.
- 14 tables with indexes, RLS, and multi-tenant `hospital_id` separation.
- 13 incremental migrations covering culture tables, master reference tables (3 iterations), audit fields, cascade deletes, frequency alerts, and timeline enrichment.

### 2. Authentication (JWT, not Supabase Auth)
- Backend: Express REST API with `jsonwebtoken`. `POST /api/auth/login`, `POST /api/auth/signup`, `GET /api/auth/me`.
- Frontend: `AuthContext` stores user + token in platform storage. Login/signup screens redirect with auth gate.
- Passwords hashed with SHA-256 (demo-grade; production should use bcrypt).
- Seed credentials: `meera@stjohn.icu` / `admin123`, `arjun@stjohn.icu` / `consult123`, `isha@stjohn.icu` / `resident123`.

### 3. Frontend Data Layer
- `DataContext` loads all entities via REST API on mount (9 concurrent calls + dashboard + timeline).
- API client created once at module level (backend uses `service_role` key, no auth needed for data).
- All screens use `useData()` instead of mock arrays.
- No loading spinners — content renders immediately, empty states handle missing data.

### 4. Tab Navigation (Web Fix)
- `(tabs)/_layout.js` uses `useState` + direct component imports + `display: none` instead of Expo Router's `Tabs`.
- Reason: Expo Router Tabs on web causes full page reloads when switching tabs.
- All screens stay mounted (state preserved), only visibility toggles.

### 5. UI Polish
- Dashboard: Fixed magic `+1` on positive cultures; `Updated` reads from `hospital.updatedAt`.
- Settings: Shows actual logged-in user via `useAuth().user`, not `users[0]`.
- Investigation/antibiotic detail: Removed filler placeholder text; stub action buttons show honest alerts instead of navigating to wrong screens.
- New resource form: Save button has loading state (disabled + ActivityIndicator + "Saving...").
- Search: Shows "No results found" empty state.
- Auth fetch: 10-second AbortController timeout (avoids silent hangs).

### Critical Weaknesses (P0 — Must Fix Before Any Production Use)

1. ⚠️ **No auth on 50+ CRUD endpoints** — anyone can read/write/delete all patient data
2. ⚠️ **Unsalted SHA-256 password hashing** — not bcrypt; passwords crack instantly
3. ⚠️ **Service role key exposed** — committed to git; full database admin access
4. ⚠️ **Mass assignment vulnerability** — raw `req.body` passed to DB on insert/update
5. ⚠️ **Hardcoded JWT secret fallback** (`'icu-steward-dev-secret'`) in `backend/src/config/env.js`
6. ⚠️ **Open CORS** — no origin restriction on any endpoint
7. ⚠️ **Error messages leak internal details** — SQL errors, stack traces returned verbatim
8. ⚠️ **No rate limiting, no request size limits** — brute force and DoS possible
9. ⚠️ **Services layer fully instantiated but entirely unused** — ~200 lines of dead code
10. ⚠️ **Orphaned dead code** (`backend/src/server.js`) — references nonexistent files
11. ⚠️ **Schema mismatches** between service layer and database tables (notifications, audit_logs)
12. ⚠️ **Role config keys mismatch seed data roles** — RBAC cannot match

### 4-Phase Remediation Plan

#### Phase 1: Immediate Fixes (Week 1) — P0, must fix before ANY production use

**Step 1.1: Add authentication to all CRUD endpoints**
- `backend/src/app.js`: Apply `authenticateJWT` middleware globally or wrap all route mounts
- `backend/src/middleware/build-repo-routes.js`: Add auth check as first middleware in each route definition
- `backend/src/middleware/auth-middleware.js`: Verify the middleware extracts `req.user` correctly
- Expected: Every `/api/*` endpoint (except login/signup) requires a valid JWT Bearer token

**Step 1.2: Replace SHA-256 with bcrypt**
- Run: `yarn backend add bcryptjs` (or `npm install bcryptjs` in backend/)
- `backend/src/modules/auth/auth.routes.js`:
  - Replace `crypto.createHash('sha256').update(password).digest('hex')` with `bcryptjs.hashSync(password, 10)` on signup (line 33)
  - Replace hash comparison with `bcryptjs.compareSync(password, user.password_hash)` on login (line 47)
- `backend/supabase/master.sql` and seed data: Replace SHA-256 hashes with bcrypt hashes of same passwords
- Expected: Passwords stored as bcrypt hashes with salt; login verifies via bcrypt compare

**Step 1.3: Fix secrets management**
- Create/update `.gitignore` at repo root and `backend/.gitignore` to include `.env`
- Run: `git rm --cached backend/.env` to remove tracked `.env` file
- `backend/src/config/env.js` (line 14): Remove `|| 'icu-steward-dev-secret'` fallback — throw error if `JWT_SECRET` is unset
- Rotate all Supabase keys (URL, anon key, service role key) and JWT secret via Supabase dashboard
- Expected: `.env` never committed; server fails to start if `JWT_SECRET` is missing

**Step 1.4: Add input validation / prevent mass assignment**
- Install validation library: `yarn backend add zod` (or `joi`)
- `backend/src/middleware/build-repo-routes.js`:
  - Add `validate(schema)` middleware before `repo.insert(req.body)` and `repo.update(req.params.id, req.body)`
  - Create per-entity schemas in `backend/src/validation/` (e.g., `patient.schema.js`, `investigation.schema.js`)
  - Whitelist only allowed fields per entity for insert/update
- Expected: Only validated, whitelisted fields reach the database

**Step 1.5: Restrict CORS**
- `backend/src/app.js` (line 24):
  - Change `app.use(cors())` to `app.use(cors({ origin: ['http://localhost:8081', 'https://aiims-icu.onrender.com'] }))`
  - Load allowed origins from `CORS_ORIGINS` env var for flexibility
- Expected: API rejects requests from unknown origins

**Step 1.6: Add rate limiting and request size limits**
- Run: `yarn backend add express-rate-limit`
- `backend/src/app.js`:
  - Add `app.use(express.json({ limit: '1mb' }))` (line 25)
  - Add global rate limiter: `app.use(rateLimit({ windowMs: 15 * 60 * 1000, max: 100 }))`
  - Add stricter limiter on `/api/auth/login`: `max: 10` per 15 minutes
- Expected: API rejects oversized payloads and rate-limited requests

## Notes

- ⚠️ **Security is the #1 priority**. The entire CRUD API is publicly accessible with no auth. Do not deploy anywhere until Phase 1 remediation is complete.
- **JavaScript only** (no TypeScript anywhere).
- **Yarn v1 only** (run `yarn dev` from root for both servers).
- **Node 20.x required.**
- **Backend is ESM** (`"type": "module"`), all local imports need `.js` extensions.
- **Backend `.env` requires** `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `JWT_SECRET`. (After Phase 1, `SUPABASE_SERVICE_ROLE_KEY` should be removed in favor of `SUPABASE_ANON_KEY` + RLS.)
- **Frontend `yarn dev` uses `CI=1`** to skip interactive Expo prompts.
- **No test framework configured** — use `vitest` for backend, React Native Testing Library for frontend (recommended in Phase 2).
- **Dead code present**: `backend/src/server.js` references nonexistent files and would crash if executed. `backend/src/services/*.js` are instantiated but never wired into routes. Delete or wire them during remediation.
- **Schema mismatches exist**: `notifications` table is missing columns expected by `notification-router.js`; `audit_logs` service passes `payload` key but schema expects `details`. Fix in Phase 2 Step 2.1.
- **Role config keys** (`admin`, `doctor`, `nurse`) in `backend/src/config/roles.js` do not match seed data role names (`Hospital Admin`, `ICU Consultant`, `Senior Resident`). Fix in Phase 2 Step 2.4.
- **Frontend hardcodes** `ACTIVE_HOSPITAL_ID = 'hosp-st-john'` and `ACTIVE_USER_ID = 'u-admin'` in `src/lib/config.js`. After auth is enforced, derive these from the authenticated user's JWT at runtime.
